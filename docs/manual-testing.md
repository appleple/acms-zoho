## 2. a-blog cms 側の動作確認（実画面）

### 2-1. 前提（これが無いと1件も送信されない）

1. `.env` に `HOOK_ENABLE=1`
2. docker 起動済み、`pnpm build` 済み（`app/dist/acms-zoho.js` が最新）
3. **Zoho OAuth 認証が完了していること** — 拡張アプリ管理画面（`admin/app_zoho_index`）で認証。未認証だとタブ/項目のプルダウンが出ず、送信処理も即終了する。

> UI のバッジ・パネル・ツールチップ等は **a-blog cms 管理画面本体の CSS が読み込まれた実画面**でスタイルが当たる（`mock/*.html` は本体CSSが無いため見た目は当たらない）。

### 2-2. 最小の連携設定（スモークテスト：見込み客へ1件 insert）

対象フォームの「Zoho Form 設定」を開いて設定する。

- **Zoho CRM連携**: 有効（チェック）
- **送信ルール**（1行）※現在は「詳細UI」のみ
  - 新しく登録するZohoタブ: `見込み客 (Leads)`
  - 既存を書き換えるZohoタブ: **空のまま** → 動作バッジが「追加のみ」になる
  - 同じレコードと判断する項目: 追加のみのため使用しません（自動で無効表示）
  - → 動作サマリに「💡 送信のたびに『見込み客』へ新規レコードを作成します。」と出る
- **Zohoと連携するフィールドの設定**（1行）
  - 種別: フィールド
  - a-blog cms のフィールド名: `name`（実フォームのフィールド名に合わせる）
  - タブ名: `見込み客`
  - 項目名: `姓`（Last_Name。**「必須」バッジが付く**。Zoho 見込み客の必須項目なので必ずマップ）
  - 追加: ON / 更新: OFF
- **Zoho リレーショナル設定**: 空のまま

→ `name` フィールドを持つフォームを送信すると、Zoho の見込み客（Leads）に新規レコードが1件作成される。

### 2-4. upsert / 横断も試す場合（README 実例に近い構成）

- 送信ルール: 新しく登録するZohoタブ=`見込み客`、既存を書き換えるZohoタブ=`見込み客`＋`連絡先`、同じレコードと判断する項目=`Email`
  → 動作バッジが「追加＋更新」、サマリが「💡『Email』で『見込み客・連絡先』の既存を探し、あれば更新・なければ『見込み客』へ新規作成します。」になる
- フィールド設定に `email`→`メール`、`company`→`会社` などを追加

### 2-5. テストケース集（いろいろな設定を試す）

**事前準備**
- テスト用フォームに `name` / `email` / `company` / `tel` / `inquiry` などのフィールドを用意しておく
- Zoho 側に「既存確認用」として、あるメールアドレス（例 `dup@example.com`）の**見込み客**と**連絡先**を1件ずつ作っておくと、更新・upsert・横断が試せる
- 各ケースの後は、Zoho CRM 側で作成/更新されたレコードと、失敗時はログ（§3）を確認する

## 3. ログの見方（送信結果の確認）

- 成功: `Logger::info`「データの一括送信が完了しました」
- 失敗: `Logger::error`（管理画面 UI にはエラーは出ない）。例: `MANDATORY_NOT_FOUND`（必須項目未設定）
- `isDebugMode()`（`DEBUG_MODE`）を有効にすると例外がそのまま throw されデバッグしやすい

## 4. 注意点（既存仕様・今回は変更していない）

- `modulePriority = ['Contacts','Leads']` はコード固定。見込み客/連絡先は「検索して追加/更新振り分け（横断）」ロジックに乗る。
- 「追加＋更新（upsert）」は非優先モジュール（商談など）では実質 insert になる場合がある（`RecordApi` の実装）。
- Zoho 側でタブ/項目を追加・変更したら「↻ Zohoから再取得」でキャッシュをクリア（既定 TTL 1時間）。
- 送信ルールは複数タブの「横断」を保持するため詳細UIのみ（1タブ＋動作セグメントのシンプル版は互換性の都合で不採用）。
